# Chaos UI 组件库规范与样式布局深度审计报告

> 基于只读调研（规范文档 / 导出面 / 组件实现 / designs 看板 / 测试与 Storybook）。**不改代码**，先产出结论与整改路线。

---

## 一、总体结论

项目已有**较完整的书面规范**（`CONVENTIONS.md`、`component-spec.md`、`AGENTS.md`、`designs/` 流程）和**大规模组件库存**（UI ~183 / Business ~330 / Layout ~27 / Mobile ~27）。

主要问题不是“没有规范”，而是：

1. **规范未真正绑定代码**（文档、ESLint、覆盖率、Story play 与现状脱节）
2. **公共 API 不诚实**（主入口 vs README 示例）
3. **样式/布局产品化债务**（Admin 壳层不一致、z-index 失控、双实现）
4. **API 一致性不足**（size 体系、DateRangePicker 双份、Loading 双概念）

v1.3–1.5 已还清一批硬债（shadow package、假 QR、AdminShell 脱 AppShell、Tab 双实现合并）。**剩余是产品化与规范落地债。**

## 二、不符合 UI 组件库规范的问题（按严重度）

### 🔴 HIGH — 必须优先处理

| #   | 问题                                   | 证据                                                                          | 为何不合规                                        |
| --- | -------------------------------------- | ----------------------------------------------------------------------------- | ------------------------------------------------- |
| H1  | **主入口文档与运行时不一致**           | `tsup` 将 `.` → UI barrel；README 示例却从根导入 `DataTable` 等 business 组件 | 消费者按文档安装即失败；专业库最基本的 API 诚实性 |

| H4  | **Story 交互约定 0 落地**              | CONVENTIONS/AGENTS 要求 `play`；515 个 story **零** `play`                    | 组件库交互契约缺失                                |
| H5  | **ESLint `no-missing-story` 路径过时** | 规则找 `src/components`，实际在 `src/stories/*`                               | 质量门禁假阳性/假阴性                             |

### 🟠 MEDIUM — 影响可维护性与消费体验

| #   | 问题                             | 说明                                                                                         |
| --- | -------------------------------- | -------------------------------------------------------------------------------------------- |
| M1  | Story/Test 矩阵不全              | UI 缺 ~38 tests / ~41 stories；layout 仍有洞                                                 |
| M2  | Token 文档与 `styles.css` 不同步 | 文档写 `--color-primary` / `globals.css`，实际是 `--primary` / `styles.css`                  |

### 🟡 LOW — 技术债/可延后

- `COMPONENT_INDEX` 计数过期
- 无 design-token 包 / Figma 流水线 / 多品牌主题
- 无 API Extractor 导出冻结
- 扁平目录 500+ 文件可发现性差
- 2.0 前保留的 casing 别名噪声

## 三、样式 / 布局深度问题清单

### S0 — 产品级布局风险

#### 1. Admin 壳层体系不一致

| 项           | 现状                                                                                         |
| ------------ | -------------------------------------------------------------------------------------------- |
| 高度         | `AdminHeader` = `h-16`；`TopBar`/`AppShell` = `h-14`（差 8px）                               |
| 折叠宽       | `AdminShell`/`AdminSider` 默认 64；`AppShell` 默认 0                                         |
| 通知         | AdminShell 用 `NotificationCenter`；AdminHeader 仍暴露 `notificationCount` 铃铛 API — 双路径 |
| 移动端 aside | AdminShell 的 aside `hidden lg:block`，**小屏直接消失无 drawer 回退**                        |
| 双壳         | `AppShell` 与 `AdminShell` 并存且职责边界文档不清                                            |

文件：`components/layout/admin-shell.tsx`、`admin-header.tsx`、`admin-sider.tsx`、`app-shell.tsx`

#### 2. z-index 失控

| 来源                   | 值                                                |
| ---------------------- | ------------------------------------------------- |
| Dialog/Sheet 等        | `z-50`                                            |
| UI GlobalLoading       | `z-[100]`                                         |
| 设计 token             | `--z-index-*` = **1000–1080**（几乎未被组件使用） |
| Tour / ImageViewer     | `z-[9998/9999]`                                   |
| Business GlobalLoading | `zIndex=9999`                                     |

结果：Loading / Modal / Tour / Lightbox 可互相遮挡，无统一堆叠策略。

### S1 — 明确的样式/行为缺陷与重复

| 问题                          | 详情                                                                                                                                |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| **GlobalLoading 双重文案**    | `Spinner label={text}`（sr-only + aria）+ 可见 `<span>{text}</span>` → 重复播报/冗余                                                |
| **Mobile 双栈**               | `ui/mobile-pull-refresh` vs `mobile/mobile-pull-to-refresh`；`ui/mobile-swipe-action` vs `mobile/mobile-swipe-actions` — API 不兼容 |
| **Table 产品线混乱**          | `Table` → `DataTable` → `SearchTable` → `ProTable` + 仍导出的 deprecated `AdvancedDataTable`                                        |
| **QrCodeDisplay 缺迁移 warn** | 设计计划写 dev `console.warn`，实现仅是 thin wrapper，无 warn                                                                       |
| **QR 默认色无视主题**         | `fgColor="#000" bgColor="#fff"`                                                                                                     |

### S2 — Magic number / 硬编码 / 主题一致性

- Shell 宽度硬编码：`240 / 64 / 280`
- Picker 列表 `max-h-64/72` 约 70 处，无密度 token
- 树缩进 `12 + level * 16` / `level * 16`
- 代码/日志表面：`bg-zinc-950`、`text-zinc-*`（亮色主题下突兀）
- 遮罩透明度混用：`bg-black/10`、`/40`、`/50`、`/80`、`/90`
- 图表/业务状态大量硬编码 hex / Tailwind 调色板

### S3 — 可接受但需文档化的分层

- `Spin`（包裹内容）/ `Spinner`（图标）/ `GlobalLoading`（遮罩）— 意图分层 OK
- `Notification*` 四级 — 设计计划明确不合并
- `EmptyState` vs `MobileEmptyState` 薄包装

## 四、API 一致性快照（影响组合使用）

| 模式                 | 健康度 | 说明                                                                        |
| -------------------- | ------ | --------------------------------------------------------------------------- |
| `cn()` + `data-slot` | ✅ 好  | UI 主流一致                                                                 |
| Base UI 原语         | ✅ 好  | 已从 shadcn/Radix 迁移大半                                                  |
| `className` 透传     | ✅ 好  | 少数如 StatusTag 偏弱                                                       |
| size 体系            | ❌ 差  | 多套词汇表                                                                  |
| loading              | ⚠️     | Button 缺；业务工具栏有                                                     |
| ref                  | ⚠️     | 仅少数 form 控件 forwardRef                                                 |

| 表单                 | ⚠️     | RHF Form* / LabeledField / SchemaForm 三套心智                              |

## 六、建议整改路线（分 4 波，先文档计划再编码）

按 `AGENTS.md`：**没有计划文档 = 不能开始开发**。建议先落一份：

`designs/2026-07-11_组件库规范与样式布局审计整改_设计计划.md`

### Wave 1 — 规范诚实性 + 消费 API（1–2 天）

1. 修正 README / ARCHITECTURE 主入口说明与示例（根 = UI only；business/layout 走 subpath）
2. 补全 exports 表（`/layout`、`/mobile`、`/business/server`）
3. 对齐 Storybook 端口、AI prompts 路径、token 文档与 `styles.css`
4. 修复 ESLint `no-missing-story` 路径；评估 `require-forward-ref` 是否适配 React 19
5. DateRangePicker / GlobalLoading **命名与文档边界**先定（实现可 Wave 2）

### Wave 2 — S0 布局与堆叠（2–3 天）

1. **Shell 统一策略**：header 高度 token、collapsed 默认值、通知单一路径、AdminShell 移动端 aside（Sheet/Drawer）
2. **z-index 政策**：token → Tailwind utility；Dialog/Sheet/Tour/GlobalLoading/ImageViewer 全量迁入；禁止新 `z-[9999]`
3. GlobalLoading：去掉双重 label；补 `role="status"` / `aria-live`；z 对齐 modal 层

### Wave 3 — 双实现与 API 梯子（2–3 天）

1. Mobile kit：选定 `components/mobile/*` 为 pull/swipe 规范源；ui 变体 re-export 或 deprecate
2. Table 梯子文档 + 隐藏/弱化 AdvancedDataTable 发现路径
3. size 标准草案：`sm | default | lg`（business `md` → `default`）
4. Button `loading`（或官方组合文档）
5. 废弃组件补 dev `console.warn`（QrCodeDisplay、MultiTabManager、AdvancedDataTable）

### Wave 4 — 质量门禁与主题消费（持续）

1. 覆盖率阶梯：60 → 70 → 80 → 85，与 CONVENTIONS 对齐
2. 交互 primitives 强制 Default + variants + disabled + 1 个 `play`
3. axe 矩阵扩到全部 UI primitives
4. 业务侧硬编码色 → semantic token 批量替换（status/progress/chart 映射表）
5. PeerDeps 按 feature 可选化（core / table / chart / editor / dnd）
