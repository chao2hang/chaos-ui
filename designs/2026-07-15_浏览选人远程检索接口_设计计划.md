# 浏览 / 选人远程检索接口 设计计划

## 概述

补齐 **组件库侧检索契约**：浏览框 / 选人组件统一把 `keyword` 交给消费方 `loadData` / `loadUsers` / `fetcher`，**不实现拼音**；后端负责 `zc` / `zhangchao` →「章超」等匹配。

## 背景与动机

- 库已有 `BrowseDialog`、`RemoteSelect` 远程通道，但 `UserBrowse` / `UserPicker` / `EmployeePicker` 仍以本地数组 + `includes` 为主。
- 消费方需要稳定接口接 HR API，而不是在库内做拼音。

## 设计方案

### 原则

1. **keyword 原样下发** — 不做本地拼音展开。
2. **远程优先** — 提供 `loadUsers` / `fetcher` 时，列表数据以远程结果为准；本地 `users`/`options` 仅作初始/兜底（按组件文档约定）。
3. **非 breaking** — 新 prop 均为可选；仅本地数组时行为与现网一致。
4. **与 BrowseDialog 对齐** — 搜索防抖、loading、空态；`BrowseLoadParams.keyword` 文档标明后端检索职责。

### 接口设计

```typescript
// UserBrowse / UserPicker
loadUsers?: (params: { keyword: string }) => Promise<User[] | UserItem[]>;
searchDebounceMs?: number; // default 300

// EmployeePicker
fetcher?: (keyword: string) => Promise<EmployeePickerOption[]>;
debounceMs?: number; // default 300

// BrowseDialog / RemoteSelect — 已有，仅加强 JSDoc
loadData?: (params: BrowseLoadParams) => Promise<BrowseLoadResult<T>>;
// params.keyword: 原样搜索词；匹配（含拼音）由后端完成
fetcher?: (keyword: string) => Promise<RemoteOption[]>;
```
