# 组件深度检测修正计划 设计计划

## 概述

基于对 546 个注册组件的自动化检测和人工抽样检测结果，制定系统性修正计划。修正覆盖五大问题域：源码路径元数据错误、模板描述、代码示例缺失、API/Props 文档缺失、示例代码质量不足。

## 检测结果摘要

### 自动化检测统计

| 指标 | 数量 | 占比 |
|------|------|------|
| 总组件数 | 546 | 100% |
| Grade A (85-100) | 60 | 11.0% |
| Grade B (70-84) | 191 | 35.0% |
| Grade C (50-69) | 295 | 54.0% |
| Grade D (<50) | 0 | 0.0% |

### 问题分布

| 问题类型 | 数量 | 主要分布 |
|----------|------|----------|
| 模板描述 | 267 | Business: 267 |
| 无真实代码示例 | 426 | Business: 328, DataDisplay: 26, System Layout: 23, Form: 21, Feedback: 11, General: 7, Layout: 6, Navigation: 4 |
| 无真实 Props 表格 | 113 | 全分类 |
| 源码路径错误 | 256 | Business: 233, General: 10, System Layout: 7, Form: 4, Feedback: 2 |

### 人工抽样发现

| 分类 | 抽样组件 | 发现 |
|------|----------|------|
| General | Icon, Spin, Typography, Button | v1.0.7 已修复，质量良好 |
| Form | Input, Select | Input 代码示例仅 `<Input />` 无 props；Select 第二个示例错误 |
| DataDisplay | Table | 有真实代码示例，但 API 表格仅 className + ...rest |
| Feedback | Alert, Dialog | Alert 质量好（有 variant 示例）；Dialog API 表格不完整 |
| Navigation | Breadcrumb | 代码示例不完整（BreadcrumbList 内缺少子组件） |
| Business | AccountBalance, AmountInput 等 | 模板描述、无真实代码示例、sourcePath 指向错误位置 |

### 核心问题

1. **sourcePath 元数据错误（256 个）**：`components.meta.ts` 中 `sourcePath` 指向 `packages/chaos-design-ui/components/...`，但实际文件在 `components/...` 下
2. **模板描述（267 个）**：Business 分类组件统一使用 `"面向 ERP/企业场景的业务组件"` 模板文案
3. **代码示例无效（426 个）**：`<Component />` 裸实例化，无 props 传递
4. **API/Props 表格模板化（113+ 个）**：仅 `className` + `...rest`，未从源码提取真实 props
5. **部分代码示例错误**：如 Select 的 `<Select>Normal</Select>` 不符合组件 API

## 修正方案

### 优先级

```
P0（阻塞性，先修）
├── 修正 sourcePath 元数据（256 个组件）
└── 修正错误代码示例（如 Select 的不合规用法）

P1（质量提升，分批修）
├── 替换 267 个模板描述为真实描述
├── 补写 426 个组件的真实代码示例
└── 完善 API/Props 表格

P2（锦上添花）
├── 补充变体/尺寸/状态示例
└── 增加真实业务场景示例
```

### 技术方案

#### 修正 1：sourcePath 元数据修复

编写脚本 `scripts/fix-source-paths.mjs`：
- 遍历 `components.meta.ts` 中所有组件
- 检查 `sourcePath` 指向的文件是否存在
- 如果 `packages/chaos-design-ui/` 下不存在，检查 `components/` 下是否存在
- 自动修正 `sourcePath` 为正确路径

#### 修正 2：模板描述替换

对 267 个 Business 分类组件：
- 读取组件源码，提取组件功能信息
- 生成真实的中文和英文描述
- 更新 `components.meta.ts` 中的 `desc` / `descZh`

策略：编写脚本自动提取源码注释和组件名，生成初始描述，然后人工审核。

#### 修正 3：代码示例补写

对 426 个无真实代码示例的组件：
- 读取源码 Props 接口
- 生成包含核心 props 的基础用法示例
- 对于复合组件（Dialog, Select 等），生成完整的组合示例
- 对于 Business 组件，生成包含 mock 数据的示例

策略：编写脚本 `scripts/generate-code-examples.mjs` 自动生成基础示例模板，然后人工精修。

#### 修正 4：API/Props 表格完善

对所有仅有 `className` + `...rest` 的组件：
- 从源码提取 Props 接口定义
- 对于继承原生元素的组件（如 Input extends `input`），列出最常用的原生属性
- 对于继承 @base-ui/react 的组件，列出库提供的关键 props
- 生成完整的 API 表格

策略：扩展 `component-audit.mjs` 提取 props，生成 Markdown 表格模板。

#### 修正 5：错误代码示例修复

手动修正已知错误示例：
- Select: `<Select>Normal</Select>` → 使用正确的复合组件用法
- Breadcrumb: 补完 BreadcrumbItem/BreadcrumbLink 子组件
- 其他类似问题

## 风险与注意

1. **组件量大（546 个）** — 脚本自动化处理元数据和基础示例，人工聚焦质量审核
2. **sourcePath 修正可能影响构建** — 修正后需验证 `generate-component-loader.mjs` 正常工作
3. **Business 组件 mock 数据** — 部分组件需要 mock 数据才能展示，示例代码需包含 mock
4. **i18n 一致性** — 修改 .zh.mdx 后必须重新运行 split-mdx-locales.mjs 生成 .en.mdx
5. **不要修改 components.meta.ts 结构** — 仅修改 desc/descZh/sourcePath 字段值
