# 组件细节一致性检查与整改 设计计划

## 概述

针对「看起来像组件坏了、其实是细节未对齐设计系统」类问题，建立**可重复的检查清单 + 分波整改**。以 AddressInput（原生 `<select>` 未走 `NativeSelect`、高度/外观与 `Input` 不一致、Story 示例区划不在内置选项）为样板，覆盖整库同类细节，避免只修单点。

**本计划不做**：大功能新增、API 大爆炸、文档站全站重构（已有独立计划）。

## 背景与动机

### 为什么需要？

- 文档站 / Storybook 用户反馈「样式有问题」，多数根因在**组件实现细节**，而非预览壳。
- 库内已有对齐 primitives（`Input` `h-8`、`Select`/`NativeSelect`、`Spinner` 等），业务/复合组件仍大量手写原生控件与临时代 class。
- 已完成的「组件库规范与样式布局审计」偏壳层 / API / 双实现；**表单控件细节一致性**未单独立项。

### 已核实样板（AddressInput，2026-07-12）

| 现象                   | 根因                                       | 处置（已修，作对照）              |
| ---------------------- | ------------------------------------------ | --------------------------------- |
| 省市区像浏览器默认下拉 | 手写 `<select>`，无 `appearance-none`/箭头 | 改 `NativeSelect`                 |
| 与街道 Input 高度不齐  | select `h-9` vs Input `h-8`                | 统一 default `h-8`                |
| 三列挤/缩              | NativeSelect 外层曾 `inline-flex`          | 外层 `flex w-full min-w-0`        |
| Story 区县空白         | 示例「浦东新区」不在内置 `DISTRICTS`       | 示例改库内真实值                  |
| 预览区过窄             | Preview 居中无 `w-full` 约束               | 预览 chrome 加 `max-w-3xl w-full` |

### 扫描快照（计划创建时）

- 业务/UI 中仍含原生 **`<select`** 的源文件约 **24**（排除 `native-select.tsx` 自身与 calendar 测试）：
  - business：`form-dialog`、`filter-bar`、`stock-transfer-dialog`、`meeting-room-booking`、`multi-currency-input`、`notification-rule-builder`、`report-builder`、`dashboard-designer` 等
  - ui：`phone-input`、`editable-table`、`editable-descriptions`、`spreadsheet-editor`、`help-desk`、`audio-player` 等
- 手写 `h-9 … border … select` 一类 class 片段约 **14+** 处，与 default Input/Select 梯子不一致。

### 预期效果

1. 表单类控件在默认 size 下视觉与 `Input`/`Select`/`NativeSelect` 一致。
2. 新代码默认走 primitives；原生 `<select>` 仅允许在白名单（性能/表格密集/特殊 embedding）并写注释。
3. Story / 文档预览的 **value 必须落在 options**；预览壳不挤压表单布局。
4. 有**脚本化检查**可重复跑，不必靠肉眼点详情页。

## 设计方案

### 分层模型

```
L0 规范（写清楚）
  component-spec / CONVENTIONS：高度梯子、何时 NativeSelect vs Select、禁止裸 select
L1 自动检查（可 CI）
  rg / 小型 node 脚本 / ESLint no-restricted-syntax（可选）
L2 人工抽检清单（详情页 + Storybook）
  表单 / 弹层 / 表格内嵌 / 移动端
L3 分波整改
  P0 高频表单 → P1 业务编辑器 → P2 重型设计器/表格内嵌
```

### 统一规则（检查与整改共用）

#### R1 — 选择器 primitive

| 场景                                   | 必须用                          | 禁止                                     |
| -------------------------------------- | ------------------------------- | ---------------------------------------- |
| 单选、需 a11y 弹层、选项多             | `Select` + `SelectTrigger`…     | 裸 `<select>` 套临时代 class             |
| 移动端 / SSR / 级联简单 / 表格行内轻量 | `NativeSelect`                  | 无箭头、无 `appearance-none` 的裸 select |
| 多选                                   | `MultiSelect` / 既有 multi 组件 | 手写 multi native                        |

**例外白名单（须行内注释 `// native-select-exception: <reason>`）**：

- 超密集 spreadsheet 单元格且已证明 Select 成本不可接受
- 第三方嵌入强制 native
- 测试专用 mock

#### R2 — 默认高度梯子（desktop）

| 角色              | class / data-size   | 说明                                                             |
| ----------------- | ------------------- | ---------------------------------------------------------------- |
| default 输入/触发 | `h-8`               | 与 `Input`、`SelectTrigger` default、`NativeSelect` default 一致 |
| sm                | `h-7`               | 表格行内、紧凑工具条                                             |
| lg                | `h-9` 或组件既有 lg | **不得**把 default 写成 `h-9` 冒充标准表单                       |

#### R3 — Token / class

- 边框：`border-input`；背景：`bg-transparent` / `dark:bg-input/30`；圆角：`rounded-lg`（与 Input 族一致，避免无故 `rounded-md` 混用在同一表单行）
- 禁用：`disabled:opacity-50 disabled:cursor-not-allowed`
- 焦点：`focus-visible:border-ring focus-visible:ring-ring/50 focus-visible:ring-3`
- 禁止：无 token 的 `border-gray-*` / 随意 `zinc-*`（与既有 token 审计一致）

#### R4 — 布局

- 表单根：`flex flex-col gap-*` + `w-full min-w-0`
- 多列：`grid grid-cols-1 … sm:grid-cols-N`，子项控件 `w-full`
- 包装 `relative inline-flex` 的控件在 grid 内必须可撑满（`w-full`）

#### R5 — Story / 文档数据诚实

- 受控 `value` 必须属于 options / 内置字典
- 假数据优先用组件内 sample 常量，禁止「看起来真、选项对不上」
- 业务态 Story 至少：空态 + 已填态；可选 disabled / 校验

#### R6 — 预览壳（docs）

- Preview 内容区：`w-full min-w-0` + 合理 `max-w-*`（表单建议 `max-w-3xl`）
- 禁止仅 `items-center justify-center` 导致宽表单被挤成内容宽

### 技术方案（整改时）

1. **优先替换为 `NativeSelect`**：API 接近（`value`/`onChange`/`options`/`disabled`/`aria-label`），迁移成本低。
2. **需要搜索/虚拟滚动/复杂 item**：升 `Select` 组合，不硬撑 native。
3. **PhoneInput 等复合控件**：区号 select 统一 `NativeSelect` size=`sm` 或共享 trigger class token。
4. **不改公开 props 形状**，除非某组件文档已承诺错误行为（走 changelog）。
5. **测试**：现有 getByLabelText('…') 对 native/NativeSelect 均适用；替换后跑对应 `*.test.tsx`。
6. **Story**：同步改示例 value；补 `WithDisabled` / 已填态若缺失。

## 风险与注意

| 风险                                                                    | 缓解                                                                           |
| ----------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| 一次改 24 文件 diff 过大                                                | 严格 P0→P1→P2；每波可单独 PR                                                   |
| NativeSelect placeholder 用 disabled 空 option 与旧「可再选空」行为差异 | 迁移时保留 `{ value:"", label:"请选择…" }` 可选项，勿一律 disabled placeholder |
| 表格内 Select 弹层 overflow                                             | P2 优先 NativeSelect 或已有 cell editor 模式                                   |
| ESLint 一刀切红片                                                       | 先脚本 warn，再 lint error                                                     |
| 与消费方胶水计划冲突                                                    | 不改 Form 数据契约，只换呈现 primitive                                         |
| dist / docs 预览不同源                                                  | 改 `components/**` 后确认 tsup watch；story 走 `@/components` 源码             |
