# 组件深度检测 设计计划

## 概述

对 Chaos UI 全部 546 个注册组件进行系统性深度检测，覆盖五大维度：**组件功能完善性**、**样式规范符合度**、**交互逻辑正确性**、**文档齐全度**、**示例代码齐全度**。产出每组件的检测报告和修复清单，为后续迭代提供优先级排序。

## 背景与动机

### 核心问题

#### 问题 1：大量 bootstrap MDX 内容模板化

约 400+ 个 Business 分类组件的 MDX 文件仍使用模板内容：
- 描述：`"XXX — 面向 ERP/企业场景的业务组件。"`
- 何时使用：`"适用于XXX相关场景。"`
- 代码示例：`<Component />` 裸实例化，无有效 props
- API 表格：仅 `className` + `...rest`，无真实 props 文档

#### 问题 2：预览质量参差不齐

预览渲染管道三级 fallback：
1. **手写 demo**（56 个）— 质量高，但覆盖面仅 10%
2. **Story 预览**（487 个）— 依赖 Story 质量，部分 Story 仅渲染空壳
3. **自动加载器**（319 个）— 裸 `<Component />`，无 children/props 时白屏
4. **PreviewFallback** — "No live preview available"

#### 问题 3：组件功能/样式/交互未知

- 部分组件可能存在必填 props 未在预览/示例中传递 → 渲染崩溃
- 暗色模式适配未验证
- RTL 布局适配未验证
- 键盘可访问性未系统测试
- 响应式断点行为未验证

#### 问题 4：API/Props 文档缺失

大量组件的 Props 表格仅包含 `className` 和 `...rest`，未从源码中提取真实 props 定义。

## 检测维度与评分标准

### 维度 1：组件功能完善性（满分 25）

| 检查项 | 分值 | 判定标准 |
|--------|------|----------|
| 渲染正常 | 10 | 组件能无报错渲染，不白屏、不崩溃 |
| Props 可用 | 8 | 核心 props 传值后行为正确（非全 undefined） |
| 状态管理 | 4 | 有状态的组件（受控/非受控）能正常工作 |
| 边界处理 | 3 | 空数据/极端值不崩溃，有空状态展示 |

**评级**：A（22-25）| B（18-21）| C（12-17）| D（< 12）

### 维度 2：样式规范符合度（满分 20）

| 检查项 | 分值 | 判定标准 |
|--------|------|----------|
| Design Token 使用 | 6 | 使用 `border-border`、`bg-card` 等 token，无硬编码颜色 |
| 暗色模式 | 5 | dark 模式下文字/背景/边框可读，无对比度问题 |
| 响应式 | 4 | 移动端不溢出，桌面端不变形 |
| RTL 适配 | 3 | RTL 模式下布局正确（使用逻辑属性） |
| 间距/圆角一致性 | 2 | 符合 Tailwind 间距/圆角规范 |

**评级**：A（17-20）| B（14-16）| C（10-13）| D（< 10）

### 维度 3：交互逻辑正确性（满分 25）

| 检查项 | 分值 | 判定标准 |
|--------|------|----------|
| 点击/触发 | 6 | 按钮/链接/可点击元素响应正确 |
| 键盘可访问性 | 6 | Tab 可聚焦，Enter/Space 触发，Esc 关闭 |
| 焦点管理 | 5 | 弹层类组件打开/关闭后焦点正确转移 |
| ARIA 属性 | 4 | 有正确的 `role`、`aria-label`、`aria-expanded` 等 |
| 防抖/节流 | 4 | 输入类组件有合理防抖，无性能问题 |

**评级**：A（22-25）| B（18-21）| C（12-17）| D（< 12）

### 维度 4：文档齐全度（满分 15）

| 检查项 | 分值 | 判定标准 |
|--------|------|----------|
| 组件描述 | 3 | 有真实的、非模板的功能描述（中英文） |
| 何时使用 | 3 | 有具体的使用场景说明 |
| API/Props 表格 | 5 | 列出全部公开 props，含类型、默认值、说明 |
| 注意事项 | 2 | 标注 Client/Server、已知限制、依赖说明 |
| i18n 双语 | 2 | `.en.mdx` 和 `.zh.mdx` 内容完整且语义正确 |

**评级**：A（13-15）| B（10-12）| C（7-9）| D（< 7）

### 维度 5：示例代码齐全度（满分 15）

| 检查项 | 分值 | 判定标准 |
|--------|------|----------|
| 基础用法 | 4 | 有可运行的基础示例代码 |
| 关键变体 | 4 | 展示 size/variant/disabled 等关键属性 |
| 真实场景 | 4 | 至少 1 个贴近实际业务场景的示例 |
| 代码可运行 | 3 | import 路径正确，props 类型匹配，无 TS 报错 |

**评级**：A（13-15）| B（10-12）| C（7-9）| D（< 7）

### 综合评级

| 总分 | 评级 | 含义 |
|------|------|------|
| 85-100 | A | 生产就绪 |
| 70-84 | B | 基本可用，有小问题 |
| 50-69 | C | 需要修复 |
| < 50 | D | 严重问题，阻塞性 |

## 技术方案

### 检测工具链

#### 工具 1：自动化脚本 `scripts/component-audit.mjs`

编写 Node.js 脚本，批量检测：
- **静态分析**：解析组件源码 AST，提取 props 定义，与 MDX 中 API 表格对比
- **MDX 质量扫描**：检测模板文案、空代码示例、缺失章节
- **预览覆盖率统计**：对比 `components.meta.ts` 与三个预览源，输出覆盖矩阵
- **i18n 一致性检查**：对比 `.en.mdx` 和 `.zh.mdx` 章节结构是否一致

输出：`docs/audit/component-audit-report.json` + `docs/audit/component-audit-summary.md`

#### 工具 2：Playwright 视觉检测脚本 `scripts/visual-audit.cjs`

已有脚本扩展：
- 遍历全部 546 个组件页面（`/components/[category]/[slug]`）
- 浅色 + 深色模式各截屏一次
- 检测控制台错误/警告
- 检测预览区域是否白屏（EmptyRenderSensor 触发）
- 输出：`output/playwright/visual-audit/` 目录下的截图 + JSON 报告

#### 工具 3：Storybook 交互测试

利用现有 Storybook stories：
- 运行 `pnpm run test-storybook` 检查 Story 交互测试
- 对缺失交互测试的 Story 标记为待补

#### 工具 4：人工检测清单模板

为每批次提供标准化检测清单（见下方实施步骤），检测人员按清单逐项检查。

### 检测流程

```
Phase 0：自动化扫描（工具 1 + 工具 2）
  ↓ 产出全量报告 + 优先级排序
Phase 1-8：按分类逐批人工深度检测
  ↓ 每批产出修复清单
Phase 9：汇总报告 + 修复优先级矩阵
```

## 单组件检测清单模板

每个组件的检测遵循以下标准流程：

### 1. 功能检测

```
□ 打开 /components/{category}/{slug} 页面
□ 检查预览区域是否正常渲染（非白屏、非 Fallback、非 Error）
□ 检查控制台是否有 error/warning
□ 切换到 Storybook 查看完整 Story（如有）
□ 阅读源码，确认 Props 接口定义
□ 验证核心 Props 传值后行为是否正确
□ 检查受控/非受控模式是否正常
□ 检查空数据/极端值是否崩溃
```

### 2. 样式检测

```
□ 检查是否使用 Design Token（无硬编码 #color）
□ 切换暗色模式，检查可读性
□ 缩小窗口至 375px，检查响应式
□ 切换 RTL 模式（如有 Direction wrapper），检查布局
□ 检查间距/圆角是否符合 Tailwind 规范
□ 检查 hover/focus/active 状态样式
```

### 3. 交互检测

```
□ 点击主要交互元素，验证响应
□ Tab 键遍历，检查聚焦顺序
□ Enter/Space 触发主操作
□ Esc 关闭弹层（如有）
□ 检查 ARIA 属性（role/aria-label/aria-expanded）
□ 检查弹层焦点管理（打开聚焦、关闭还原）
□ 快速连续操作，检查防抖/节流
```

### 4. 文档检测

```
□ 检查 .zh.mdx 描述是否真实（非模板）
□ 检查 .en.mdx 描述是否对应
□ 检查"何时使用"章节是否有具体场景
□ 检查 API/Props 表格是否完整（对照源码 Props 接口）
□ 检查注意事项章节
□ 检查 i18n 双语内容是否语义一致
```

### 5. 示例代码检测

```
□ 检查是否有"基础用法"代码示例
□ 检查 import 路径是否正确
□ 检查代码中的 props 是否与源码 Props 接口匹配
□ 检查是否有变体示例（size/variant/disabled）
□ 检查是否有真实场景示例
□ 尝试复制代码到沙箱运行（抽查）
```

## 风险与注意

1. **组件量大（546 个）** — 自动化扫描覆盖全量，人工检测分批进行，优先处理 P0 问题
2. **Business 分类组件量大（346 个）** — 分 6 批处理，每批约 50-60 个
3. **Playwright 截图耗时** — 546 组件 × 2 主题 = 1092 张截图，预计 30-60 分钟
4. **部分组件需运行时数据** — Business 组件依赖 mock 数据，预览可能无法完全验证
5. **Storybook 交互测试覆盖率** — 部分 Story 缺少交互测试（play function），不影响检测但需标注
6. **暗色模式检测** — 需要文档站点支持暗色切换（通过 `next-themes`）
7. **RTL 检测** — 需要使用 `<Direction dir="rtl">` 包裹，部分组件可能未适配
