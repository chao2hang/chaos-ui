# Component Guidelines

Standards and best practices for building Chaos UI components.

## Architecture

### Component Layers

```
components/
├── ui/           # Base UI components (atomic)
├── business/     # Business domain components
└── layout/       # Page-level layouts
```

### Layer Rules

1. **UI Components** (`components/ui/`)
   - Base, reusable components
   - No business logic
   - Highly customizable

2. **Business Components** (`components/business/`)
   - Built on top of UI components
   - Contains domain-specific logic
   - Examples: DataTable, FormField, StatusTag

3. **Layout Components** (`components/layout/`)
   - Page-level layouts
   - Compose business and UI components

## Naming Conventions

| Type | Convention | Example |
|------|-----------|---------|
| Files | `kebab-case.tsx` | `status-tag.tsx` |
| Components | `PascalCase` | `StatusTag` |
| Props Types | `ComponentNameProps` | `StatusTagProps` |
| CSS Variables | `--kebab-case` | `--brand-500` |
| Hooks | `use` prefix | `use-mobile.ts` |
| Constants | `UPPER_SNAKE_CASE` | `ORDER_STATUSES` |

## Component Structure

### Basic Component Template

```tsx
"use client"

import * as React from "react"
import { cva, type VariantProps } from "class-variance-authority"
import { cn } from "@/lib/utils"

// 1. Variant definitions
const componentVariants = cva(
  "base-classes",
  {
    variants: { ... },
    defaultVariants: { ... },
  }
)

// 2. Props interface
interface ComponentNameProps
  extends React.HTMLAttributes<HTMLDivElement>,
    VariantProps<typeof componentVariants> {
  // Custom props
}

// 3. Component implementation
function ComponentName({ ... }: ComponentNameProps) {
  return <div className={cn(componentVariants({ ... }))} />
}

// 4. Exports
export { ComponentName, componentVariants }
export type { ComponentNameProps }
```

## Design Token Usage

Always use semantic tokens, never hardcoded colors:

```tsx
// ✅ Correct
<div className="bg-primary text-primary-foreground" />

// ❌ Incorrect
<div className="bg-blue-500 text-white" />
```

### Status Color Pattern

```tsx
// For badges and tags
className="bg-success/15 text-success"
className="bg-warning/15 text-warning"
className="bg-destructive/10 text-destructive"
```

## State Management

### Focus States

```tsx
"focus-visible:border-ring focus-visible:ring-3 focus-visible:ring-ring/50"
```

### Disabled States

```tsx
"disabled:pointer-events-none disabled:opacity-50"
```

### Validation States

```tsx
"aria-invalid:border-destructive aria-invalid:ring-3 aria-invalid:ring-destructive/20"
```

## Dark Mode

All components must support dark mode:

```tsx
// Background colors
"dark:bg-input/30"  // Default
"dark:bg-input/50"  // Hover

// Border colors
"dark:border-input"

// Invalid states
"dark:aria-invalid:border-destructive/50"
```

## Data Attributes

Every component should use data attributes for state and variant:

```tsx
// Slot identification
<div data-slot="component-name" />

// Size variant
<div data-size={size} />

// Visual variant
<div data-variant={variant} />

// State
<div data-state="open" />
```

## Accessibility

1. All interactive elements must be keyboard accessible
2. Use `aria-label` for icon-only buttons
3. Use `aria-describedby` for form field descriptions
4. Use semantic HTML elements
5. Maintain 4.5:1 contrast ratio for text
6. Use `sr-only` for screen reader context

## Performance

1. Use `"use client"` ONLY when needed
2. Prefer Server Components by default
3. Use `React.memo()` for expensive renders
4. Use `useCallback` for event handlers passed as props
5. Use `next/dynamic` for heavy components

## Testing

1. Every component should have a Story
2. Include all variants in stories
3. Use `argTypes` for complete Props documentation
4. Tag with `autodocs` for automatic documentation
5. Add `tags: ['a11y']` for accessibility testing

## Anti-Patterns

Avoid these patterns:

- ❌ Raw hex colors in className
- ❌ Inline styles for static values
- ❌ `any` type
- ❌ Modifying UI components directly
- ❌ `console.log` in production
- ❌ Hardcoded strings (use constants)
- ❌ Missing TypeScript types
- ❌ Using `@apply` when a Tailwind class exists
- ❌ Importing from relative paths (use `@/` alias)
